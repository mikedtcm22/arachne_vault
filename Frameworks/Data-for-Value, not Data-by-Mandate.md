# Data-for-Value, not Data-by-Mandate

Up: [[Arachne Intelligence]] · Part of [[Finance-as-a-Service]]

The FaaS stance on how Finance acquires the upstream data it runs on: by **exchange, not authority**.

## The analogy (why)
Finance runs on information the way a startup runs on capital — the more and better the data, the more and better Finance can build. A startup doesn't *mandate* its cash flow; it ships a product that solves a problem and receives capital in exchange. FaaS acquires data the same way: give a contributing function a tool that makes *their* work easier, and the data they maintain in return is the price they're glad to pay.

## The concept (what)
**"Data-for-value, not data-by-mandate."** Authority-based collection degrades the moment enforcement attention lapses — the contributor has no stake in quality. Exchange-based collection stays clean because the contributor benefits directly. This is *why* FaaS treats adoption as a measurable success criterion rather than an assumption.

## The reciprocal-value test (how)
For any Finance system that depends on another department maintaining upstream data quality, ask: *"How does this department giving Finance the data let Finance hand* them *something that makes* their *job easier?"*
- A report merely *generated from* their data fails the test (a dashboard exists, but it doesn't help them do their own work).
- A passing mechanism returns a tool that helps them with a job they already own.
- **Worked example (ResFrac pricing):** Engineering maintains granular per-customer cloud-spend tagging → Finance uses it for margin/pricing work → Finance returns a usage-forecast that improves Engineering's reserved-instance planning (lower compute cost they already own). They maintain the tagging because it lowers their bill, not because Finance asked.
